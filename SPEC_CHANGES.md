## 2. jGrants API — 実仕様の確定（推測でなくOpenAPI＋実測で確認）

- OpenAPI yaml を取得し確認: `jgrants-api.yaml`（デジタル庁 developers サイト掲載）
- ベースURL: `https://api.jgrants-portal.go.jp/exp`
- 一覧API `/v1/public/subsidies` の**必須パラメータは `keyword`（2文字以上）/ `sort` / `order` / `acceptance`**。
  「全件取得」はできない仕様のため、**JAPONISM関連キーワードを複数回検索して ID で重複排除する方式**（キーワードスイープ）を採用。
- 一覧レスポンスには URL・概要・利用目的が**含まれない**ため、`/v2/public/subsidies/id/{id}` 詳細APIを
  件別に呼び出す（1秒間隔・ローカルキャッシュで再取得を回避）。
- 詳細APIの `front_subsidy_detail_page_url` が公式詳細ページURL（`https://www.jgrants-portal.go.jp/subsidy/{id}`）。
- `acceptance=1`（募集期間内のみ）で取得。レート制限の公表なし → 自主的に1リクエスト/秒に制限。

## 3. JETRO — RSSを主とせず、一覧ページ2枚をパースする方式に確定

- RSSは直近約20件のみで網羅性がない → **新着検知の補助**として利用。
- 本体は `events/tradefair.html`（国内外の展示会・商談会一覧。ステータス[出展募集]等・会期・開催地がテーブルで取れる）と
  `services/tradefair/list.html`（ジェトロ出展支援の年間予定）の2枚をスクレイピング。
- robots.txt を確認: Disallow 指定は特定クローラー（WiSEWebCrawler）向けのみで、一般アクセスの禁止パスに該当しない。
  それでも**リクエスト間隔3秒・日次1回・取得結果キャッシュ**で負荷を最小化する。

## 4. J-GoodTech の扱い変更

仕様書では自動収集対象だったが、実確認の結果、ページがJavaScriptレンダリングで
静的取得では内容が得られず、商談会情報の閲覧自体も会員向け機能が中心。
→ **自動収集の対象から外し、手動フォールバック（スプレッドシート/CSV）経由で登録する運用に変更。**
（J-GoodTech自体への会員登録・活用は営業施策として別途推奨）

## 5. 主要展示会（主催者サイト）— 個別スクレイパーではなくキュレーション方式

ギフトショー等の主催者サイトは構造がバラバラでスクレイパーが壊れやすい。
→ Phase 2 では **`sources/curated_exhibitions.json`（人手で検証した主要展示会リスト＋公式URL＋会期）** を
コレクターの一つとして読み込む方式に変更。会期・締切は公式サイトで確認した値のみを記載し、
年次更新はこのファイル1枚の編集で済む。取りこぼしは手動シートで補完（仕様書の思想どおり「自動を主、手動を保険」）。

## 8. スキーマの微修正

仕様書 第6章のスキーマに以下を追加（既存フィールドは変更なし）:

- `summary`: 案件概要の短文（サイトのカード表示用。jGrantsは詳細HTMLをテキスト化して先頭を抜粋）
- `status`: `open / upcoming / closed / unknown`（JETROの[出展募集]/[募集終了]表記や募集期間から判定）
- `match_reasons`: スコアの根拠になったキーワード一覧（フィルタ調整・デバッグ用）

## 9. サイトのデータ読み込み方式

`data.json` の fetch は `file://` で開くとブラウザのCORS制限で失敗するため、
パイプラインが **`data.json`（ホスティング用）と `data.js`（`window.__DATA__` 代入・ローカルでダブルクリックで開く用）の両方**を出力し、
サイトは data.js → fetch の順でフォールバックする。ホスティング環境でもローカルでも同じ index.html が動く。

## 10. 実行環境まわり

- GitHub Actions cron: `0 21 * * *`（UTC 21:00 = JST 06:00）＋ `workflow_dispatch`
- 各コレクターは独立実行・try/except で失敗を握りつぶし、実行ログに「取得件数/フィルタ後件数/失敗ソース」を出力（仕様書どおり）
- コレクターが失敗した場合は前回の raw データ（`data/raw/`）を温存し、成功分だけ更新する

## 11. ホスティングの確定（2026-07-03 ユーザー決定）

- **GitHub Pages に確定**（ユーザー選択）。仕様書第9章の推奨（Cloudflare Pages + Access）ではなく、
  手軽さを優先。**URLを知っていれば誰でも閲覧できる公開サイトになる**点は選択時に説明済み。
  緩和策として `<meta name="robots" content="noindex, nofollow">` を設定済み（検索エンジンには載せない）。
- デプロイは `update.yml` 内で収集→コミット→公開を同一ワークフローで行う。
  ユーザー自身のコードプッシュ時は収集をスキップして公開のみ実行。
- **2026-07-04追記**: actions/deploy-pages（workflow方式）が初回成功以降、コンテンツ非依存で
  `deployment_failed` を連続で返したため（GitHub側キューで即時失敗・10分後の再試行も失敗）、
  **gh-pagesブランチ配信方式（Source: Deploy from a branch）に変更**。site/ を gh-pages ブランチへ
  force pushする方式で、切替後は安定して配信されている。

## 12. イベントの業種違い除外ルール追加（2026-07-03 ユーザーフィードバック）

「商談会に食品専門など無関係な案件が多い」との指摘を受け、展示会・商談会に限り
**タイトル・タグに業種違いの語（食品・機械・医療・シルバー産業・アニメ・繊維等）が
含まれる案件を除外**するルールを追加（`pipeline/merge_and_filter.py` の `event_fits`）。
- JETROの出展支援年間予定は全産業の見本市を含むため、このルールが特に効く（56件→16件）
- 「非食品」表記は食品除外の対象外（むしろ適合案件）として特別扱い
- 手動追加分とキュレーション分（人が選んだもの）は除外対象外
- 補助金には適用しない（販路開拓系補助金は業種横断のため）

## 13. 地域限定フィルタとソース拡充（2026-07-04 ユーザーフィードバック）

**地域限定の除外**（`region_allowed`）: 本社=東京都新宿区・製造元(OEM)=福岡のため、
「特定地域の企業限定」案件は東京・福岡関連のみ掲載する。
- 補助金: jGrantsの補助対象地域（構造化データ）が 全国/東京/福岡/海外/関東・甲信越/九州・沖縄 を含まないものは除外
- イベント: 「●●県産」「●●県内企業」「【●●市】」等の企業限定表現をタイトル・概要から検出して除外。
  **開催地が他県・海外でも企業を限定しないもの（大阪開催の商談会等）は掲載対象のまま**

**商談会ソースの拡充**（要望「商談会がもっとあるはず」への対応）:
- **東京商工会議所**（`fetch_cci.py`）: myevent.tokyo-cci.or.jp のイベント一覧（受付中約400件）から
  商談・マッチング・交流会・展示会系のみ採用。セミナー・講習会、ゴルフ・就職等の交流イベントは除外
- **東京都中小企業振興公社**（`fetch_kosha.py`）: お知らせ一覧から商談会・マッチング・販路系を採用。
  「販路拡大助成」等は補助金カテゴリに分類（都内中小企業向けでJAPONISM適合）
- **キュレーション追加**: ててて商談会2026.9（9/2-4・スパイラルホール）、
  合同展示会「大日本市」2026年9月展（9/2-4・TFTホール）— いずれも工芸・クラフトメーカー向け商談会
- **中小機構（SMRJ）のイベント一覧は不採用**: service.smrj.go.jp がJSレンダリングのため静的取得不可
- 各サイトのcharsetヘッダ欠落対策として、HTMLパースはbytes渡し（bs4のmeta自動判定）に統一
